# UI 设计稿说明

本文件夹是 `markdown_viewer` 的设计源真相(source of truth)。实现(`Sources/MarkdownViewer/main.swift`)对齐时,**一切以这里为准**。

## 文件清单与定位

| 文件 | 性质 | 作用 |
|---|---|---|
| `Markdown Viewer.dc.html` | **终稿(成品稿)** | app 真实那一屏的完整样子:布局 + 行为 JS + 每个元素的精确内联样式。**这是权威。** |
| `Design System.dc.html` | 抽象规范(设计系统) | 通用 token / 原则手册:颜色、圆角阶梯(4/6/8/10/14/99)、阴影分档、字号阶梯、"安静优先·琥珀只一处"等原则。**不是某一屏的具体实现。** |
| `Markdown Viewer Icon 1024.png` | 资源 | app 图标源图(1024×1024),用于生成 `AppIcon.icns`。 |
| `support.js` | 预览运行时 | 仅供 `.dc.html` 在浏览器里预览,不参与 app。 |

## ⚖️ 优先级规则(裁决铁律)

1. **终稿 `Markdown Viewer.dc.html` 永远优先。** 某个元素只要画在了 app 屏幕上,就以终稿的具体数值/样式为准。
2. **设计系统只在"终稿未注明"处生效。** 终稿没覆盖到的地方(终稿里不存在的元素、未给出的状态/尺寸),才参照 `Design System.dc.html` 的 token 与原则**自由发挥**。
3. **两者数值冲突时 → 跟终稿,不跟设计系统。** 设计系统给的是取整/示意值,终稿会对具体元素做微调,微调后的终稿值才是最终值。

> 类比:Design System 是"字典和语法书",Markdown Viewer 是"用这套语言写成的那篇文章"。读文章看文章本身,字典只在文章没说清时查。

## 已知冲突(均以终稿为准)

对稿时容易踩的几处——左列(终稿)正确,右列(设计系统示意)**不要采用**:

| 元素 | ✅ 终稿(采用) | ❌ 设计系统示意(忽略) |
|---|---|---|
| 标签 tab | 高 28 / 圆角 6(终稿 L102) | 组件卡 高 32 / 圆角 8 |
| 查找面板阴影 | `0 8px 28px rgba(0,0,0,.14)`(终稿 L135) | 电梯档 `0 6px 24px /.12` |
| 状态栏文字色 | `#767676`(终稿) | 色板 `#8e8e93` |
| 查找开关芯片 Aa/W/.* | `22×22 圆角6`(终稿 L143–145) | 圆角阶梯把 4 标作 "chip" |

> 教训:外部 review 若拿"设计系统示意值"来挑实现的毛病,多半是误报——先回查终稿对应行号再下结论。

## 给实现/核验者

- 核验先打开**终稿**逐元素比对内联样式,不要只看 Design System 色卡。
- CSS → AppKit 换算注意:`box-shadow` 的 CSS blur ≈ CALayer `shadowRadius` 的 2 倍(blur 28px ≈ radius 14);CSS 向下偏移 `0 8px` ≈ CALayer `shadowOffset.height = -8`(CALayer 负 y 向下)。
- 琥珀色 `#E8A33D` 仅限终稿明示处:未保存点、查找命中、当前大纲项、拖入高亮(终稿 L220 的 inset 边框)。其余一律中性灰。
